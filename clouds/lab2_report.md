# Лабораторная работа №2

В лабораторной раоте №1 в IT Towers были выделены категории (области ИТ) Облачных сервисов, DevOps и Хранилищ. При работе с биллингами сервисов Microsoft Azure из области DevOps сервисов не встретилось, так как не использовались никакие технологии для сборки и интеграции срвисов, но это не нарушило принципы, которым следовали коллеги при работе с Amazon Web Services, так как сервисная модель (классификатор) является унифицированным шаблоном, описывающим возможные категории затрат. Данные биллинга отражают лишь **фактическое** потребление за конкретный период. Отсутствие строк с потреблением DevOps-сервисов означает лишь нулевое использование ресурсов этой категории в данном отчетном периоде, что не повреждает целостность кросс-платформенной модели.

## Ход работы
Для начала я решила изучить, что за зверь этот ваш Microsoft Azure (мое скептическое отношение к продуктам этой компании заложено в генах, даже на компе стоит Ubuntu, но оказалось, что вещь довольно интересная, хоть и мудреная :)). Если коротко, Microsoft Azure - облачная платформа, предоставляющая более 200 продуктов и облачных служб, которые позволяют разрабатывать, развертывать и управлять приложениями в глобальной сети центров обработки данных. Azure поддерживает модели обслуживания IaaS, PaaS и SaaS, обеспечивая полный цикл разработки ПО.

### IT Tower: Cloud Services
Основная категория, вобравшая в себя большую часть вычислительных и платформенных сервисов.
- `Service Family: Analytics` (как для Amazon Redshift): Использована для Azure Analysis Services (аналитика данных в облаке, OLAP-кубы). Это полностью управляемая PaaS - облачная платформа для сбора сырых данных из разных источников, приведения их к единому виду для дальнейшего использования в отчетах и визуализациях, а также благодаря унификации данных делает их безопасными для дальнейших операций.

- `Service Family: Application Integration` (как для Amazon SNS): Использована для Azure Data Factory v2 (интеграция и перемещение данных) и Azure Scheduler (планирование задач). **Azure Data Factory v2** используют для миграции данных, например, из реляционных БД в нереляционные, при этом с сохранением необходимых связей и зависимостей. Сервис поддерживает создание ETL/ELT процессов (извлечение, трансформация и загрузка), при этом не требует глубокого знания языков программирования, так как обладает довольно нативно понятным интерфейсом. **Azure Scheduler** - это планировщик задач, собственно, использовали его для автоматизации запуска процессов по расписанию или через конкретные промежутки (вполне практично для бэкапов). На данный момент сервис выведен из эксплуатации и заменен на **Azure Logic Apps**.

- `Service Family: Compute Services` (новая подкатегория): Введена для Azure Virtual Machines и Azure Batch. В исходных данных AWS «чистые» вычисления (EC2) отсутствовали (был только Redshift), поэтому для виртуальных машин Azure потребовалось отдельное семейство. **Azure Virtual Machines** предоставляет разнообразные виртуальные машины (IaaS-ресурсы) с различными параметрами, каждый точно сможет найти что-то по своим потребностям. **Azure Batch**, в свою очередь, позволяет резервировать ресурсы и запускать довольно ресурсоемкие процессы в облаке. Резервация заключается в том, что сервис сам выделяет необходимые мощности для выполнения желаемых задач и сам управляет их выполнением, а пользователь просто платит за время использования.

- `Service Family: Databases` (новая подкатегория): Введена для управляемых баз данных Azure Database for MySQL и Azure Database for PostgreSQL. Тут много не сказать, так как названия говорят сами за себя: реляционные базы данных для работы, конечно же, с данными (PaaS) :)

- `Service Family: Security and Identity` (как для AWS Directory Service): Использована для Azure Key Vault (управление ключами) и Azure Sentinel (системы безопасности). **Azure Key Vault** позволяет избежать хардкодинга и изолировать такие важные данные, как пароли, ключи шифрования, сертификаты и так далее. Сервис позволяет централизованно управлять секртеными данными,удалять, записывать и изменять их. **Azure Sentinel** создан для централизованного анализа процессов, происходящих во всей системе в целом, что часто применяется для кнтроля утечки данных и подозрительных действий.

### IT Tower: Storage
Категория для сервисов хранения, кэширования и доставки контента.

- `Service Family: Caching` (новая подкатегория): Использована для Azure Cache for Redis (управляемый кэш). Этот сервис предоставляет пользователям возможность хранить частоиспользуемые данные в оперативной памяти сервиса, за счет чего доступ к этим данным осуществляется в разы быстрее, что ускоряет работу и других сервисов.

- `Service Family: Storage&Content Delivery` (как для Amazon S3/Glacier): Использована для Azure CDN (доставка контента). Этот сервис создан для того, чтобы доступ к данным имели пользователи серверов, разбросанных по разным точкам мира. Это позволяет снизить нагрузку и вместо одного сервера хранения использовать несколько, причем они будут более легкодоступными с точки зрения запросов.
  
- `Service Family: Data Transfer` (новая подкатегория): Использована для Azure Data Box (физический перенос данных). Этот сервис предоставляет пользователям возможность хранить большие объемы данных (от 80 ТБ) в облаке Azure с использованием защищенных физических устройств, пересылаемых службой доставки, так как передача по сети неэффективна.


## Вывод

В ходе лабораторной работы была успешно проанализирована структура биллинга облачного провайдера Microsoft Azure и сопоставлена с ранее изученной моделью Amazon Web Services (AWS).

1) Создана единая кросс-провайдерная модель: Мне удалось адаптировать иерархическую структуру классификации затрат (IT Tower → Service Family), разработанную для AWS, к сервисам Azure. Это подтвердило универсальность выбранного подхода: несмотря на различия в названиях продуктов (например, Amazon Redshift и Azure Analysis Services), их функциональное назначение и место в инфраструктуре идентичны.

2) Выявлены различия в потреблении: Были обнаружены категории сервисов, специфичные для конкретного набора данных Azure (например, Compute Services для виртуальных машин и Databases для управляемых БД), что потребовало расширения модели новыми семействами сервисов без нарушения общей логики.

3) Достигнута вендор-независимость: Работа показала, что, абстрагируясь от маркетинговых названий конкретного вендора и оперируя функциональными категориями (Вычисления, Хранение, Интеграция), можно строить прозрачную аналитику для мультиоблачных сред.

Таким образом, цель работы достигнута: сформировано комплексное видение облачной инфраструктуры, где сервисы разных провайдеров рассматриваются как взаимозаменяемые функциональные блоки единой IT-системы.
